CNCF's Graduated P2P file distribution project Dragonfly now speaks Hugging Face and ModelScope natively. Two backends merged into the dfget client add `hf://` and `modelscope://` URL schemes that handle authentication, revision pinning, and recursive fetches. The CNCF blog republished the announcement on Monday, putting it in front of platform teams who'd missed the original drop on d7y.io.

The math is the headline. Pulling a 130 GB model across 200 GPU nodes used to mean 26 TB of egress hitting the Hugging Face origin. With Dragonfly's seed peer in front, that drops to roughly 130 GB. A 99.5% reduction.

## How it looks in practice

The new commands are about as simple as you'd hope:

```sh
dfget hf://deepseek-ai/DeepSeek-R1/model.safetensors -O /models/
dfget hf://owner/repo -O ./repo/ -r
dfget hf://owner/private-model/weights.bin --hf-token=hf_xxx
dfget hf://deepseek-ai/DeepSeek-R1/model.safetensors --hf-revision v2.0
```

Same pattern for ModelScope with `--ms-token` and `--ms-revision`. You don't need to mirror the hub, manually shard files, or build a custom CSI plugin to get models onto nodes. dfget understands the URL scheme, hits the origin once, and seeds peers as soon as the first piece lands.

That last bit matters. Dragonfly's seed peer doesn't wait to finish downloading the entire model before sharing. As soon as a single piece is ready, it can flow to peers. For a multi-hundred-gigabyte LLM, that turns hours of cold-start delay into something closer to a parallel download.

## The cold-start problem this solves

If you're running inference on Kubernetes with dozens of pods spinning up across GPU nodes, the model pull is often the slowest and most expensive part of cold start. Egress from public hubs costs real money. Bandwidth on shared corporate networks gets ugly fast.

Dragonfly graduated from CNCF in October 2025, and Hugging Face has been collaborating with the d7y team since 2023. What's new is that the protocol support now lives inside dfget itself. You stop maintaining a sync job. You stop fighting with proxy configs. Point your inference job at `hf://` and let the P2P mesh handle the rest.

## What's still missing

The blog post is light on empirical benchmarks. The 99.5% number is a traffic-reduction calculation based on file size, not a measured wall-clock improvement on a real cluster. P2P performance depends heavily on node-to-node bandwidth, which varies wildly between bare-metal, public cloud, and cross-AZ topologies. If you're evaluating this for production, run it on your own setup before celebrating.

That said, if your platform team has been writing custom code to mirror models from Hugging Face into an internal registry, this is your chance to delete a service. Try the new schemes against a single inference workload and measure cold-start time. The numbers should speak for themselves.
