Kubescape 4.0 dropped at KubeCon Europe 2026 in Amsterdam, and it's the biggest release the project has shipped since joining CNCF as a sandbox project back in 2022. The headline: runtime threat detection is now GA, the architecture got simpler, and there's a whole new surface area for scanning AI agents running in your clusters.

## Runtime Threat Detection Goes GA

The runtime engine monitors processes, Linux capabilities, syscalls, network and HTTP events, and filesystem activity. Detection rules are written in CEL (Common Expression Language) and managed as Kubernetes CRDs, so they fit naturally into GitOps workflows.

ARMO's Ben Hirschberg and the maintainer team claim the approach cuts CVE noise by over 95%. That's a bold number, but the mechanism makes sense: instead of flagging every known vulnerability in an image, the engine learns normal workload behavior through ApplicationProfiles and only alerts on deviations. Alerts forward to AlertManager, SIEM tools, Syslog, or HTTP webhooks.

The storage layer is worth noting too. Kubescape now uses Kubernetes Aggregated API to store security metadata (ApplicationProfiles, SBOMs, vulnerability manifests) separately from etcd. That matters if you're running high-density clusters where etcd is already under pressure.

## Fewer Moving Parts

Previous versions required a host-sensor DaemonSet and a separate host-agent. Both are gone. Kubescape 4.0 runs a single node-agent per node. Fewer components means fewer things to audit and fewer things that break during upgrades.

## AI Agents Get Their Own Security Controls

This is the part that's genuinely new territory. Kubescape 4.0 introduces 15 Rego-based controls covering 42 security-critical configuration points specifically for KAgent, the CNCF Sandbox project that orchestrates AI agents on Kubernetes using Model Context Protocol.

Think about what KAgent does: it creates direct pathways between AI models and your cluster's infrastructure. A misconfigured KAgent deployment could let an AI agent escalate privileges, watch namespaces it shouldn't, or hit APIs without proper network policies. Kubescape now checks for exactly these scenarios.

"We need robust security guardrails to stop agents from exploiting them for high-risk actions like unauthorized access or deleting production data," Hirschberg said. That's not hypothetical. As more teams deploy AI agents with cluster access, the attack surface is real and growing.

There's also a KAgent-native plugin that lets AI assistants query Kubescape's security data directly. So an AI agent can inspect vulnerability manifests, review RBAC configurations, and check container behavior at runtime. It's security tooling designed for both sides of the AI coin: securing agents and empowering them.

## Compliance Coverage

Kubescape 4.0 supports NSA-CISA, MITRE ATT&CK, and CIS Benchmarks (including 1.12 for vanilla Kubernetes and 1.8 for EKS and AKS). If you're already using Kubescape for compliance scanning, the runtime detection layer plugs directly into those existing frameworks.

If you're running AI workloads on Kubernetes and haven't thought about how to secure the agents themselves, this release is a good reason to start.
