Kubescape 4.0 shipped during KubeCon EU in Amsterdam, and it's the biggest release the CNCF sandbox project has put out since ARMO open-sourced the scanner back in 2021. Two things matter here: runtime threat detection is now GA, and Kubescape can scan your AI agent deployments for security misconfigurations.

## Runtime Detection, Finally Stable

The node-agent that powers Kubescape's runtime detection has been in beta for over a year. With 4.0, it's GA. The agent uses eBPF to hook into kernel-level events without requiring sidecars or kernel modules. It watches process creation, file operations, network connections, syscall patterns, and Linux capability usage across every pod on the node.

That's a lot of data. Kubescape filters it through application profiles, baseline models it builds from observed workload behavior during a learning period. Anything outside the profile triggers an alert. Think of it as an allowlist that writes itself, then flags deviations.

For teams already running Falco or Tetragon, the pitch is integration. Kubescape ties runtime alerts back to the same posture management dashboard where you're already reviewing CVE scans and RBAC misconfigurations. One pane of glass, if you're into that sort of thing.

## Scanning AI Agents on Kubernetes

This is the genuinely new part. Kubescape 4.0 ships 15 OPA Rego controls covering 42 configuration checkpoints specifically for KAgent CRDs. These controls catch the mistakes that are easy to make when you're deploying AI agents in a rush: empty security contexts, missing NetworkPolicies, controllers with cluster-wide namespace watching privileges they don't need.

It's opinionated. The controls assume you're running KAgent, which isn't the only way to deploy AI agents on Kubernetes. But it's the first time any open-source security tool has shipped dedicated checks for AI agent workloads, and the Rego policies are extensible if you're running a different agent framework.

There's also a new MCP plugin that lets AI assistants query Kubescape directly from the cluster. Your AI agent can now ask Kubescape to list vulnerability manifests, review RBAC configurations, and inspect security contexts without a human in the loop. Whether you find that reassuring or terrifying probably depends on how much you trust your agent's judgment.

## What You Should Actually Do

If you're already running Kubescape, `helm upgrade` gets you there. The runtime detection feature requires the node-agent DaemonSet, which needs `BPF` and `SYS_ADMIN` capabilities on your nodes. Check your PodSecurityAdmission policies before deploying.

If you're not running Kubescape, this release is a reasonable time to evaluate it. The combination of posture management, vulnerability scanning, and runtime detection in a single CNCF project reduces the number of security tools fighting over your cluster's resources. ARMO reports that the node-agent adds roughly 50MB of memory overhead per node, though your mileage will vary depending on workload density.

The AI agent scanning is worth trying even if you're skeptical. Run `kubescape scan framework ai-agent` against a cluster running any LLM-backed workload and see what it finds. Worst case, you'll catch a few missing NetworkPolicies you should've had anyway.
