Kubescape 4.0 dropped at KubeCon Europe in Amsterdam, and it's the kind of release that actually changes how you run the tool. Runtime Threat Detection, previously in beta, is now GA. So is Kubescape Storage, the aggregated API layer that keeps Application Profiles and SBOMs out of etcd.

The runtime engine uses CEL-based detection rules that evaluate against Application Profiles, which act as behavioral baselines for your workloads. It watches processes, syscalls, network traffic, HTTP events, and filesystem activity. When something deviates from the baseline, alerts fire to AlertManager, your SIEM, Syslog, or a plain HTTP webhook. Rules and RuleBindings are Kubernetes CRDs, so you manage them the same way you manage everything else.

Ben Hirschberg from ARMO says the storage layer "has been proven to handle the demands of large-scale, high-density clusters." That matters because earlier versions stored security metadata as standard custom resources, which could hammer etcd in clusters running thousands of workloads.

## Simpler Architecture

The old deployment model involved a host-sensor DaemonSet and a separate host-agent. Both are gone. A single node-agent per node now handles everything through direct API calls between core Kubescape microservices. Fewer privileged pods, fewer moving parts. The community had been asking for this simplification for a while.

## Scanning Your AI Agents

Here's where it gets interesting. Kubescape 4.0 ships 15 new OPA Rego-based controls covering 42 security configuration points specifically for AI agent workloads running on Kubernetes. These controls check for things like missing NetworkPolicies on agent namespaces, over-privileged namespace watching, and empty security contexts.

There's also a KAgent plugin that lets AI assistants query your cluster's security posture directly. They can inspect vulnerability manifests, review RBAC configurations, and examine runtime behavior through ApplicationProfiles and NetworkNeighborhoods. It's a two-way street: Kubescape secures your AI agents, and AI agents can use Kubescape to understand your cluster.

## Updated Compliance Baselines

CIS Benchmark coverage now includes version 1.12 for vanilla Kubernetes and version 1.8 for both EKS and AKS. If you're running compliance scans on managed clusters, that's a meaningful gap closed.

Kubescape joined the CNCF as an incubating project in January 2025, maintained primarily by ARMO. Amir Malka recently joined as a new maintainer, while David Wertenteil and Craig Box moved to emeritus status.

If you're already running Kubescape in CI for posture scanning but haven't looked at the runtime side, 4.0 makes a strong case to deploy the in-cluster operator. The runtime detection is no longer experimental, and the simplified architecture removes most of the deployment friction that kept teams from adopting it earlier.
